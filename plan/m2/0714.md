## 一、追问答案清单

### 面试官可能追问的问题及回答

#### Q1: 为什么选择端侧模型而不是云端 API？

```
回答要点：
① 低延迟——端侧推理 < 3s，云端 API 至少 1-2s 加上网络延迟
② 隐私——数据不出手机，适合处理个人敏感信息
③ 离线可用——无网络环境也能正常工作
④ 成本——没有 API 调用费用

但是端侧方案也有局限：模型能力弱于 GPT-4 等大模型，
适合工具调用这类有限场景，不适合开放域对话。
```

#### Q2: TinyLlama 1.1B 这个模型你是怎么选的？

```
回答要点：
① 参数量筛选——1B 是手机可接受的极限
   - 7B (Phi-2, Gemma-2B)：太大，手机跑不动（需要 1.5GB+）
   - 0.5B (Qwen2.5-0.5B)：太小，能力不足
   - 1.1B (TinyLlama)：刚刚好
② 量化选择——Q4_K_M 在推理速度和质量之间平衡
   - Q8：太大（1.1GB）
   - Q3：质量下降明显
   - Q4_K_M：仅 410MB，质量可接受
③ 中英文能力——TinyLlama 在多语言上表现不错
④ 社区支持——llama.cpp 官方示例直接支持
```

#### Q3: 推理延迟多少？怎么优化的？

```
回答要点：
① 当前数据
   - 模型加载：~3s
   - 单步推理：~2.5s
   - 两步任务：~5s
② 已做的优化
   - 设置线程数 = CPU 核心数
   - 控制 max_tokens = 256（限制输出长度）
   - 后台预加载模型（Application.onCreate）
③ 待做的优化
   - 换 Q3 量化（速度提升 30%）
   - 换更小的模型（Qwen2.5-0.5B，~200MB）
   - 缓存对话历史，避免重复处理
```

#### Q4: ReAct 循环中的 JSON 解析如果失败了怎么办？

```
回答要点：
① 解析流程
   - 先用正则提取文本中的 JSON 块
   - 用 org.json.JSONObject 解析
   - 如果 action 为空或 params 不合法，返回 null
② 失败处理
   - 连续 3 次失败 → 降级为纯文本回复
   - 告知用户"请换一种方式描述"
   - 推荐几种正确的指令格式
③ 为什么不是 100% 成功
   - 小模型（1.1B）的输出不稳定
   - 有时模型回到预训练时的语言习惯
   - 通过更好的 Prompt 设计可以改善
```

#### Q5: 这个项目跟 LangChain 的 Agent 有什么区别？

```
回答要点：
① 定位不同
   - LangChain 是通用的 Agent 框架，面向 Python 端
   - 我这个是 Android 端特定场景的轻量实现
② 核心思路一致
   - 都是 Tool 注册 + 模型调用 + ReAct 循环
③ Android 端的特殊考量
   - 内存受限：不能像 Python 端加载大模型
   - 系统 API 集成：需要 Android 权限管理
   - 工具类型不同：手机端是系统工具，而非 API 调用
```

#### Q6: toolManager 是怎么设计的？为什么用接口？

```
回答要点：
① 接口设计
   - AgentTool 接口：getSchema() + execute()
   - 接口的好处：解耦、可扩展、方便测试
② 注册机制
   - ToolManager 内部维护 Map<String, AgentTool>
   - register() 注册，execute() 按名称查找执行
③ 为什么不用抽象类
   - 接口更灵活：工具之间没有共享状态
   - 每个 Tool 完全独立，只有一个 execute 方法
④ 扩展性
   - 新增工具只需实现接口，调用 register()
   - 不需要修改任何现有代码（开闭原则）
```
