### 逐段话术

#### 1. 项目背景（3 分钟）

```
"这个项目的初衷是探索端侧 AI 在 Android 系统上的落地。
市面上很多 Agent 方案依赖云端 API，延迟高、有隐私风险、
离线不可用。我的目标是做一个纯端侧方案——不需要联网，
所有推理和控制在手机上完成。

我选择的切入点是系统控制——用户可以用自然语言而不是
手动操作来调用系统能力。"
```

#### 2. 技术选型（5 分钟）

```
"技术选型上，我主要考虑了三方面：

① 推理引擎：对比了 MLCEngine、MNN 和 llama.cpp。
   llama.cpp 社区最活跃，对 GGUF 格式支持最好，
   纯 C++ 实现方便 NDK 交叉编译。

② 模型：选择 TinyLlama 1.1B 是因为它小（Q4 量化后
   仅 410MB），中英文兼有，手机能跑。对比了 Qwen2.5-0.5B
   （更小但能力弱）和 Phi-2（太大跑不动）。

③ 协议设计：参考了 OpenAI Function Calling 的思路，
   用 JSON Schema 描述工具，让模型按标准格式输出。"
```

#### 3. 核心实现（10 分钟）

```
"架构上分三层：

第一层是推理层。通过 JNI 桥接 llama.cpp 的 C++ 推理引擎，
用 Kotlin 协程封装为 suspend 函数，确保推理在 IO 线程
执行不阻塞主线程。模型首次加载约 3s，后续推理单步约 2.5s。

第二层是 Tool 层。定义 AgentTool 接口，每个工具独立实现，
通过 ToolManager 统一注册和调度。用 JSON Schema 描述每个
工具的 name、description、parameters，拼接进 Prompt 让
模型了解可用能力。

第三层是 Agent 层。实现 ReAct 循环——模型输出 JSON 格式的
action + params，系统解析后执行对应 Tool，将结果以
Observation 回填到 Prompt，继续下一轮，直到输出 Final Answer。
加入 ConversationManager 维护多轮对话上下文。

错误处理：连续 3 次 JSON 解析失败则降级提示用户；
工具执行异常时回填 Observation 让模型重试；
最大 5 步循环防止死循环。"
```

#### 4. 挑战与解决（7 分钟）

```
"最大的挑战是小模型的输出不稳定。

第一个问题是输出格式不一致——模型有时不按 JSON 输出，
而是夹杂自然语言。解决：解析器兼容两种格式，先尝试提取
JSON 块，失败则降级为关键词匹配。

第二个问题是多步任务中模型遗忘前面步骤。解决：每步的
Observation 精确回填，并在 Prompt 中加入 conversation 段
记录对话历史。

第三个问题是首次加载慢。解决：在 Application.onCreate()
后台预加载模型，用户看到聊天界面时模型已就绪。"
```

#### 5. 成果与数据（3 分钟）

```
"最终效果：单步指令（如'发通知'）耗时 < 3s，
两步任务（如'打开微信并调高音量'）约 5s。
峰值内存 < 500MB，模型文件仅 410MB。

项目已开源，附带 2 分钟演示视频和详细 README。"
```

#### 6. 后续规划（2 分钟）

```
"接下来的方向：① 换更小的模型（Qwen2.5-0.5B）
降低内存占用；② 增加更多工具（截图、剪贴板）；
③ 支持语音输入。"
```
