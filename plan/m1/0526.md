## ThreadPoolExecutor 参数详解

```java
public ThreadPoolExecutor(
    int corePoolSize,      // 核心线程数
    int maximumPoolSize,   // 最大线程数
    long keepAliveTime,    // 空闲线程存活时间
    TimeUnit unit,         // 时间单位
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory threadFactory,        // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
)
```

## 各参数详细说明

### 1. **corePoolSize（核心线程数）**
- 线程池中**始终保持**的线程数量，即使它们处于空闲状态
- 除非设置了 `allowCoreThreadTimeOut(true)`

### 2. **maximumPoolSize（最大线程数）**
- 线程池中**最多允许**的线程数量
- 当任务队列满了，会创建新线程直到达到这个值

### 3. **keepAliveTime（空闲存活时间）**
- 当线程数 > corePoolSize 时，**多余的空闲线程**等待新任务的最长时间
- 超时后多余线程会被销毁

### 4. **unit（时间单位）**
```java
TimeUnit.NANOSECONDS
TimeUnit.MICROSECONDS
TimeUnit.MILLISECONDS
TimeUnit.SECONDS
TimeUnit.MINUTES
TimeUnit.HOURS
TimeUnit.DAYS
```

### 5. **workQueue（任务队列）**
| 队列类型 | 特点 | 适用场景 |
|---------|------|---------|
| `LinkedBlockingQueue` | 无界队列（默认 Integer.MAX_VALUE） | 任务量平稳，不会爆满 |
| `ArrayBlockingQueue` | 有界队列，需指定容量 | 控制资源，防止无限增长 |
| `SynchronousQueue` | 不存储任务，直接交接 | 高吞吐量，消费者充足 |
| `PriorityBlockingQueue` | 优先级队列 | 任务有优先级需求 |

### 6. **threadFactory（线程工厂）**
```java
// 自定义线程工厂示例
ThreadFactory factory = r -> {
    Thread t = new Thread(r);
    t.setName("my-pool-" + System.currentTimeMillis());
    t.setDaemon(false);  // 是否守护线程
    t.setPriority(Thread.NORM_PRIORITY);
    return t;
};
```

### 7. **handler（拒绝策略）**
当队列满且线程数达到 maximumPoolSize 时的处理：

| 策略 | 行为 |
|------|------|
| `AbortPolicy`（默认） | 抛出 `RejectedExecutionException` |
| `CallerRunsPolicy` | 由提交任务的线程自己执行 |
| `DiscardPolicy` | 直接丢弃任务，不抛异常 |
| `DiscardOldestPolicy` | 丢弃队列头部任务，重试提交 |

## 工作流程

```java
// 提交任务时的处理顺序
1. 线程数 < corePoolSize → 创建新线程执行
2. 线程数 ≥ corePoolSize → 放入队列
3. 队列满了 & 线程数 < maximumPoolSize → 创建新线程
4. 队列满了 & 线程数 = maximumPoolSize → 触发拒绝策略
```

```
提交任务
    ↓
当前线程数 < corePoolSize? ───是───→ 创建新线程执行
    │
    否
    ↓
任务队列是否已满? ───否───→ 放入队列等待
    │
    是
    ↓
当前线程数 < maximumPoolSize? ───是───→ 创建新线程执行
    │
    否
    ↓
执行拒绝策略
```

ExecutorService pool = new ThreadPoolExecutor(

ExecutorService pool = new ThreadPoolExecutor(

### 案例3：带监控的线程池
```java
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    5, 10, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100),

    System.out.println("核心线程数: " + pool.getCorePoolSize());
    System.out.println("活跃线程数: " + pool.getActiveCount());
    System.out.println("队列大小: " + pool.getQueue().size());

```

## 你代码中的参数分析

```java
new ThreadPoolExecutor(5, 5, 0L, TimeUnit.MILLISECONDS, new LinkedBlockingQueue<>())
```

| 参数 | 值 | 含义 |
|------|---|------|
| corePoolSize | 5 | 始终保持5个核心线程 |
| maximumPoolSize | 5 | 最多5个线程（不能扩容） |
| keepAliveTime | 0 | 无意义（因为大小相等） |
| workQueue | 无界队列 | 可以无限堆积任务 |

**实际效果**：
- 最多同时运行5个线程
- 任务会无限堆积在队列中，可能导致内存溢出

## 快速创建线程池的工具方法

```java
// 固定大小
ExecutorService fixed = Executors.newFixedThreadPool(5);

// 单线程
ExecutorService single = Executors.newSingleThreadExecutor();

// 可缓存（无界线程）
ExecutorService cached = Executors.newCachedThreadPool();

// 定时任务
ScheduledExecutorService scheduled = Executors.newScheduledThreadPool(5);
```

**注意**：阿里规约**不建议**使用 `Executors` 工具类创建，因为：
- `FixedThreadPool` 使用无界队列，可能 OOM
- `CachedThreadPool` 允许创建无限线程，可能 OOM

**推荐**：手动 `new ThreadPoolExecutor` 并明确参数。



import java.util.Objects;
import java.util.Random;
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class demo0525 {

//    private static int [] arr = new int[10];
//    static Semaphore mutex = new Semaphore(1);
    static BlockingQueue<Integer> arr = new ArrayBlockingQueue<>(10);
    static AtomicInteger cnt = new AtomicInteger(0);
    public static void main(String[] args) throws InterruptedException {
        ExecutorService executorService = new ThreadPoolExecutor(5, 5, 0L, TimeUnit.MILLISECONDS, new LinkedBlockingQueue<>());
        executorService.submit(() -> {
            while(true) {
                int i = new Random().nextInt(1000);
                cnt.incrementAndGet();
                arr.put(i);
                System.out.println("生产" + i);
                Thread.sleep(500);

            }
        });


        executorService.submit(() -> {
            System.out.println(Thread.currentThread().getName());
            while(true) {
                int ans = arr.take();
                cnt.decrementAndGet();
                System.out.println("消费" + ans);
                Thread.sleep(1000);
            }
        });

        executorService.shutdown();
    }
}


## 1. corePoolSize=0 时第一个任务怎么处理？

这是一个很好的边界问题！当 `corePoolSize=0` 时，处理逻辑有些特殊：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    0,  // corePoolSize = 0
    5,  // maximumPoolSize = 5
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>()
);

executor.submit(task1); // 第一个任务会怎样？
```

### 执行流程

```
第一个任务提交
    ↓
当前线程数(0) < corePoolSize(0)? → 否（相等，不创建）
    ↓
尝试放入队列
    ↓
队列成功接收任务
    ↓
检查工作线程数（发现为0）
    ↓
❗ 特殊处理：创建非核心线程（补充线程）
    ↓
新线程从队列取任务执行
```

### 源码逻辑（简化版）

```java
public void execute(Runnable command) {
    int c = ctl.get();
    
    // 1. 当前线程数 < corePoolSize？
    if (workerCountOf(c) < corePoolSize) {
        if (addWorker(command, true))  // 创建核心线程
            return;
    }
    
    // 2. 尝试放入队列
    if (isRunning(c) && workQueue.offer(command)) {
        int recheck = ctl.get();
        if (!isRunning(recheck) && remove(command))
            reject(command);
        else if (workerCountOf(recheck) == 0)
            // ⚠️ 关键：队列有任务但没有工作线程，创建非核心线程
            addWorker(null, false);
    }
    // 3. 队列满，尝试创建非核心线程
    else if (!addWorker(command, false))
        reject(command);
}
```

### 实际示例

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    0, 3, 10L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(2)  // 容量2的有界队列
);

// 提交5个任务
for (int i = 0; i < 5; i++) {
    final int taskId = i;
    executor.submit(() -> {
        System.out.println(Thread.currentThread().getName() + " 执行任务" + taskId);
        try { Thread.sleep(100); } catch (Exception e) {}
    });
}

// 执行过程：
// 任务0: 入队 → 发现无线程 → 创建线程A（非核心）执行任务0
// 任务1: 入队（队列中等待）
// 任务2: 入队（队列中等待）
// 任务3: 队列满 → 创建线程B（非核心）执行任务3
// 任务4: 队列满 → 创建线程C（非核心）执行任务4
```

## 3. Android 中为什么要使用 WorkManager？

### WorkManager 解决的问题

| 场景 | 传统方案问题 | WorkManager 方案 |
|------|------------|-----------------|
| 应用退出后执行任务 | Service 可能被杀 | 保证执行 |
| 设备重启后任务 | 任务丢失 | 自动恢复 |
| 网络不稳定的上传 | 需自己实现重试 | 内置重试+网络条件 |
| Android 版本差异 | API 不一致 | 自动选择最佳方案 |
| 省电模式 | 后台限制 | 遵循系统省电要求 |
| 周期性任务 | AlarmManager 复杂 | 简单 API |

### 核心优势

```kotlin
// 1. 保证执行（即使进程死亡、设备重启）
WorkManager.getInstance(context)
    .enqueue(OneTimeWorkRequestBuilder<UploadWorker>()
        .setConstraints(
            Constraints.Builder()
                .setRequiredNetworkType(NetworkType.CONNECTED)
                .setRequiresBatteryNotLow(true)
                .build()
        )
        .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
        .build())

// 2. 链式任务
WorkManager.getInstance(context)
    .beginWith(workA)
    .then(workB)
    .then(workC)
    .enqueue()

// 3. 唯一工作（避免重复）
WorkManager.getInstance(context)
    .enqueueUniqueWork("upload", ExistingWorkPolicy.KEEP, uploadWork)

// 4. 观察任务状态
WorkManager.getInstance(context)
    .getWorkInfoByIdLiveData(workId)
    .observe(this) { info ->
        when(info.state) {
            WorkInfo.State.SUCCEEDED -> // 成功
            WorkInfo.State.FAILED -> // 失败
            WorkInfo.State.RUNNING -> // 运行中
        }
    }
```

### API 兼容性

| Android 版本 | 后台执行方案 |
|-------------|-------------|
| API 23+ | JobScheduler（系统级） |
| API 14-22 | AlarmManager + BroadcastReceiver |
| API 26+ | 适配后台限制策略 |

WorkManager 自动选择最优方案！

### 典型使用场景

```kotlin
// 1. 上传日志（需要网络）
class LogUploadWorker(context: Context, params: WorkerParameters) 
    : Worker(context, params) {
    override fun doWork(): Result {
        return try {
            uploadLogs()
            Result.success()
        } catch (e: Exception) {
            Result.retry()  // 重试
        }
    }
}

// 2. 定期清理缓存（24小时一次）
val cleanRequest = PeriodicWorkRequestBuilder<CleanWorker>(24, TimeUnit.HOURS)
    .setInitialDelay(1, TimeUnit.HOURS)
    .build()

// 3. 应用启动时初始化（但延迟执行）
WorkManager.getInstance(context)
    .enqueue(OneTimeWorkRequestBuilder<InitWorker>()
        .setInitialDelay(5, TimeUnit.SECONDS)
        .build())
```

### 对比其他方案

| 方案 | 进程死亡 | 设备重启 | 省电优化 | 简单API |
|------|---------|---------|---------|---------|
| **WorkManager** | ✅ | ✅ | ✅ | ✅ |
| Service | ❌ | ❌ | ❌ | ❌ |
| AlarmManager | ❌ | ❌ | ❌ | 复杂 |
| JobScheduler | ✅ | ❌ | ✅ | 复杂 |
| Evernote-Job | ✅ | ❌ | ✅ | 较简单 |

**总结**：WorkManager 是 Android 官方推荐的**后台任务执行最佳实践**，特别是需要**保证执行**的任务（上报、同步、日志上传等）。
