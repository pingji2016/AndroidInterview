## Android 三大利器：LeakCanary、BlockCanary、TraceCanary

---

## 一、LeakCanary — 内存泄漏检测

### 核心原理

基于 Java **弱引用（WeakReference）** + **引用队列（ReferenceQueue）**，主动监测本该被回收的对象是否"意外存活"。

### 工作流程

```
1. 监控生命周期
   └─ ActivityLifecycleCallbacks 监听 onDestroy()
      └─ 标记为"可疑对象"，开始监控

2. 包装与观察
   └─ 创建 KeyedWeakReference 包装对象（不阻止 GC）
      └─ 关联 ReferenceQueue
         └─ 生成唯一 key，存入 Set

3. 触发 GC 并检查
   └─ 等待 5s → 调用 System.gc()
      ├─ 正常：弱引用进入 ReferenceQueue → 从 Set 移除 key → 空集，正常
      └─ 泄漏：Set 中仍包含该 key → 判定内存泄漏

4. 生成报告
   └─ Dump .hprof → 分析 GC Root 最短路径 → 通知开发者
```

### 为什么能精准定位？

弱引用指向的对象只要发生 GC 就会被回收。LeakCanary 逆向利用：若对象**应该被回收**但 GC 后弱引用**仍在**，证明有强引用持有它。

## 二、BlockCanary — 主线程卡顿检测

### 核心原理

利用 **Looper 的日志机制（Printer）**，通过监听主线程消息处理的耗时来检测卡顿。

### 工作流程

```
1. 设置 Printer
   └─ Looper.getMainLooper().setMessageLogging(printer)
      └─ 每次 Handler 处理消息前后都会调用 Printer.println()

2. 标记开始/结束
   └─ >>>>> Dispatching to Handler(h) msg=what
      └─ <<<<< Finished to Handler(h) msg=what
         └─ 记录两次调用之间的时间差

3. 判定卡顿
   └─ 时间差 > 阈值（默认 2s）
      └─ 判定为主线程卡顿（ANR 前兆）

4. 采集堆栈
   └─ 在后台线程定期 dump 主线程堆栈
      └─ 分析卡顿期间的调用栈
         └─ 定位耗时方法
```

### 代码实现

```java
// 设置 Printer
Looper.getMainLooper().setMessageLogging(new Printer() {
    private long startTime;
    
    @Override
    public void println(String x) {
        if (x.startsWith(">>>>> Dispatching")) {
            startTime = System.currentTimeMillis();
        } else if (x.startsWith("<<<<< Finished")) {
            long cost = System.currentTimeMillis() - startTime;
            if (cost > 2000) {
                // 卡顿，dump 堆栈
                dumpStackInfo(cost);
            }
        }
    }
});
```

### 关键特性

| 特性 | 说明 |
|:---|:---|
| **检测手段** | Looper Printer 前后时间差 |
| **判定阈值** | 默认 2s（可配置） |
| **堆栈采集** | 后台线程定时 `Thread.getStackTrace()` |
| **输出** | 通知栏 + Logcat + 本地日志文件 |

## 三、TraceCanary — 应用启动与页面渲染追踪

> TraceCanary 是 **Matrix**（微信开源 APM 框架）中的一部分，专注于启动优化和帧率监控。

### 核心原理

基于 **AOP（Aspect Oriented Programming）** + **字节码插桩**，在编译期自动插入埋点代码，统计关键方法的执行时间。

### 三大核心功能

#### 1. 启动耗时追踪（AppMethodBeat）

```
Application 启动
  └─ AppMethodBeat.i()  → 开始采集
     ├─ Application.onCreate()        ← 记录耗时
     ├─ Activity.onCreate()           ← 记录耗时
     ├─ Activity.onResume()           ← 记录耗时
     └─ 首帧绘制完成                  ← 冷启动结束
```

- 在编译期通过 **Transform API** 给每个方法插入时间戳记录
- 自动生成**启动耗时火焰图**，直观展示哪个阶段最慢

#### 2. 帧率监控（FPS FrameTracer）

```
每帧绘制周期
  ├─ doFrame() start
  ├─ 布局
  ├─ 绘制
  └─ doFrame() end ← 检查是否超过 16ms

超过 16ms → 记为掉帧（Jank）
  ├─ 统计掉帧次数 & 卡顿率
  └─ 记录卡顿时的堆栈信息
```

- 通过 `Choreographer.FrameCallback` 监控帧绘制时间
- 统计：**掉帧次数**、**卡顿率**、**单帧最大耗时**
- 输出：**帧率曲线图** + 卡顿堆栈

#### 3. 页面渲染追踪（ActivityLifecycleTrace）

```
Activity 启动生命周期
  ├─ onCreate
  ├─ onStart
  ├─ onResume
  └─ 首帧渲染完成

自动统计：
  ├─ onCreate → onResume 耗时
  └─ onResume → 首帧完成耗时
```

### 三款性能工具对比

| 维度 | LeakCanary | BlockCanary | TraceCanary |
|:---|:---|:---|:---|
| **检测目标** | 内存泄漏 | 主线程卡顿 | 启动耗时 / 掉帧 / 页面渲染 |
| **核心机制** | WeakReference + ReferenceQueue | Looper Printer 时间差 | AOP 字节码插桩 |
| **分析方式** | Dump .hprof → GC Root 链 | 主线程堆栈采集 | 火焰图 + 帧率曲线 |
| **阈值** | GC 后弱引用未回收 | 消息处理 > 2s | 单帧 > 16ms |
| **输出** | 泄漏路径 + 通知 | 卡顿堆栈 + 日志 | 火焰图 + 帧率统计 |
| **所属** | Square 开源 | 个人/团队项目 | 微信 Matrix |

---

## 一句话总结

- **LeakCanary**：用弱引用"钓鱼"，钓出本该回收却未回收的对象
- **BlockCanary**：在 Looper 门口蹲点，谁干活太久就记下来
- **TraceCanary**：在编译期给每个方法装上秒表，启动和渲染的每一毫秒都算清楚
