## LeakCanary 原理详解

LeakCanary 的核心原理基于 Java 的**弱引用（WeakReference）** 和**引用队列（ReferenceQueue）**，通过主动监测本该被回收的对象是否"意外存活"来发现内存泄漏。

### 工作原理（四步）

#### 1. 监控生命周期

LeakCanary 通过 `Application.ActivityLifecycleCallbacks` 等接口自动监听关键组件（Activity、Fragment）的生命周期。当某个对象的 `onDestroy` 被调用时，意味着其生命周期结束，理应被 GC 回收。LeakCanary 将其标记为"可疑对象"，开始监控。

#### 2. 包装与观察

对于每个被监控的对象（如一个 Activity），LeakCanary 执行以下操作：
- 创建一个 **`KeyedWeakReference`** 来包装它（弱引用不会阻止 GC 回收）
- 将该弱引用与一个 **`ReferenceQueue`** 关联
- 生成唯一 `key` 标识该引用，存入 `Set` 集合

#### 3. 触发 GC 并检查

等待数秒（默认 5 秒，给 GC 留足时间）后，LeakCanary 主动调用 `System.gc()` 建议 JVM 执行垃圾回收。

| 情况 | 行为 |
|:---|:---|
| **正常** | GC 回收 Activity 对象 → 弱引用进入 `ReferenceQueue` → LeakCanary 清理队列，根据引用找到对应 `key` 并从 `Set` 中移除 → `Set` 为空，一切正常 |
| **泄漏** | GC 结束后，`Set` 中仍包含该 Activity 的 `key` → 说明 Activity 虽已无用，但仍有 GC Root（如静态变量）持有强引用，导致无法回收 → 判定内存泄漏 |

#### 4. 生成分析报告

确认泄漏后，LeakCanary 执行以下步骤：
- **Dump 内存**：将 Java 堆内存快照保存为 `.hprof` 文件
- **分析引用链**：解析 `.hprof`，找出泄漏对象到 GC Root 的最短强引用路径（即泄漏源头）
- **通知开发者**：生成包含完整泄漏路径的报告，通过 Logcat 和系统通知呈现

### 为什么能精准定位？

关键在于**弱引用的特性**：弱引用指向的对象生命周期极短，只要发生 GC，无论内存是否充足，对象都会被回收。LeakCanary 逆向利用了这一点——如果一个对象**应该被回收**（生命周期结束），但 GC 后其对应的弱引用**没有被回收**，就证明该对象仍被其他强引用持有，从而精确定位内存泄漏。
