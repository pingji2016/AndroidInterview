**LeakCanary 的核心原理是基于 Java 的弱引用 (WeakReference) 和引用队列 (ReferenceQueue)，通过主动监测本该被回收的对象是否"意外存活"，来发现内存泄漏。**

它的工作流程很清晰，主要分为以下四个步骤：

### ⚙️ 工作原理四步走

1.  **监控生命周期**
    LeakCanary 会通过系统提供的 `Application.ActivityLifecycleCallbacks` 等接口，自动监听应用中关键组件（如 Activity、Fragment）的生命周期。一旦某个对象的 `onDestroy` 方法被调用，就意味着它的生命周期结束了，理应被垃圾回收器（GC）回收，LeakCanary 就会把它标记为一个"可疑对象"，开始监控。

2.  **包装与观察**
    对于每个被监控的对象（比如一个 Activity），LeakCanary 会做两件事：
    -   创建一个 **`KeyedWeakReference`** 来包装它。这个弱引用不会阻止对象被垃圾回收。
    -   将这个弱引用和一个 **`ReferenceQueue`** 关联起来。
    同时，它会生成一个唯一的 `key` 来标识这个引用，并将其存储在 `Set` 类型的集合中。

3.  **触发GC并检查**
    等待几秒钟（默认5秒，给GC一些时间）后，LeakCanary 会主动调用 `System.gc()` 建议 JVM 进行一次垃圾回收。
    -   **正常情况下**：GC 会回收该 Activity 对象，由于弱引用的特性，它会被放入关联的 `ReferenceQueue` 中。LeakCanary 清理 `ReferenceQueue` 时，会根据其中的引用找到对应的 `key`，并将它从之前存储的 `Set` 中移除。如果 `Set` 变为空，说明一切正常。
    -   **发生泄漏时**：GC 结束后，如果那个存储 `key` 的 `Set` 中仍然包含该 Activity 的 `key`，就说明**这个 Activity 虽然已经没用了，但仍有其他 GC Root 对象（比如一个静态变量）持有它的强引用，导致它无法被回收**，也就是发生了内存泄漏。

4.  **生成分析报告**
    一旦确认发生泄漏，LeakCanary 会执行以下操作：
    -   **Dump 内存**：将当前 Java 堆的内存快照保存为一个 `.hprof` 文件。
    -   **分析引用链**：解析这个 `.hprof` 文件，找出泄漏对象到 GC Root 的最短强引用路径，这正是导致泄漏的"罪魁祸首"。
    -   **通知开发者**：最终，它会生成一条包含完整泄漏路径的报告，并在 Logcat 和系统通知中呈现，帮助开发者快速定位问题代码。

### 💡 为什么它能精准定位？

关键在于利用了**弱引用**的特性：弱引用对象的生命周期很短暂，只要发生垃圾回收，无论内存是否充足，它指向的对象都会被回收。LeakCanary 正是逆向利用了这一点：如果一个对象**应该被回收**（生命周期结束），但GC后其对应的弱引用**却没有被回收**，就足以证明该对象仍然被其他强引用持有，从而确定内存泄漏。
